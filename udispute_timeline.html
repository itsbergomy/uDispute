Chapter 1

Architecture & Flow

Architecture

Two-Tier Dispute Framework

Split Round 1 disputes into a two-phase strategy modeled after how consumer attorneys actually operate.

  • Dispute Prep — 3 Notice of Dispute letters (one per bureau). Lists accounts, starts the 30-day clock. No inaccuracies revealed.
  • Bureau Assault — Detailed inaccuracy letters with FCRA citations, legal research, and e-OSCAR-resistant arguments. Sent 15-30 days later.
  • Educational panels explain each phase to the consumer

  • New flow: Confirm Account → Entity Selection → Issue/Solution Cards → Prompt Template
  • Back buttons added throughout for navigation
  • 3 route redirects changed in disputes.py

Bug Fix

Session Cookie Overflow

Full letter text stored in Flask session exceeded 4KB cookie limit. Flask silently dropped the cookie, causing redirect loops.

  • Root cause: session['tier1_letters'] held 3 full letter texts (~12KB)
  • Fix: Store only session['tier1_letter_ids'] = [14, 15, 16] (few bytes)
  • Query letters from DB on review page instead
disputes.py
Bug Fix

Inaccuracy Pre-Check Detection

Issue/solution cards weren't auto-checking based on parser findings. Debug showed detected_keys: [] even for accounts with clear issues.

  • Root cause: Parser only detected CONTRADICTIONS. Affirm Inc's status matched its negative history — no contradiction, no detection.
  • Fix: Added 3 condition-based rules that fire on scenarios (late payments, collections, charge-offs) without needing contradictions
  • False lead: Tried flask-session with SQLAlchemy backend first, then reverted

Chapter 3

Parser & Detection Engine

Parser

Contradiction-Based Detection (Rules 1-9)

Original parser detected inaccuracies only when data contradicted itself within the same report.

  • Status contradicts payment history
  • Account type mismatch
  • Original creditor not reflected
  • Closed account with balance
  • Charge-off not in status
  • Balance exceeds credit limit
  • Double/duplicative reporting
  • Missing due date / Missing payment amount
pdf_parser.py
Parser

Condition-Based Detection (Rules 10-12)

Added 3 rules that fire on universally disputable scenarios without needing contradictions. This was the breakthrough that made pre-checks work.

  • #10: Unverified late payments — any late payment entries trigger demand for date-level verification
  • #11: Unvalidated collection — any collection account triggers FDCPA debt validation demand
  • #12: Unverified charge-off — any charge-off triggers verification of charge-off date, balance breakdown, and DOFD

Chapter 4

Prompt Engineering & AI

Prompts v1

Generic System Prompts

Original prompts gave GPT vague instructions with no understanding of how disputes are actually processed.

  • "Use your knowledge of UCC, CFPB regulations..."
  • One-liner pack templates
  • No awareness of e-OSCAR, ACDV codes, or CRA processing workflows
Prompts v2

e-OSCAR Intelligence Bake-In

Researched the e-OSCAR system and injected that knowledge into every system prompt. GPT now understands the enemy.

  • _E_OSCAR_INTELLIGENCE shared block — explains 4-minute ACDV process, Response Codes (01/02/07/13/DNR)
  • 7 Letter Engineering Rules — field-specific verification, CFPB Circular 2022-07, DNR traps
  • All 4 prompt packs rewritten: default, arbitration, consumer_law, ACDV_response
  • 3 new FCRA_INACCURACY_MAP entries with e-OSCAR awareness
  • Result: Composite score jumped from 8.3 → 8.5
letter_generator.py
Prompts v3

Anti-Code-Reduction Overhaul

Autoresearch revealed letters were still being reduced to e-OSCAR codes 103 and 001 despite legal sophistication. Complete structural rewrite to resist code reduction.

  • Rule 8: Multi-point DISPUTE POINT format — forces CRA to assign multiple codes or escalate
  • Rule 9: Concrete factual allegations — "balance is $X but should be $Y" instead of generic "verify the balance"
  • Removed "2026 FCRA updates" references — undermined credibility
  • Fixed documentary demands — directed through CRA to furnisher, not demanding CRA produce docs it doesn't have
  • Cross-bureau discrepancy exploitation
  • 650-word cap to avoid frivolous-claim flag
  • Proof-of-error injection for parser-detected inaccuracies
letter_generator.py
Model

GPT-4o → o3

Upgraded the AI model from GPT-4o to o3 — a reasoning model that thinks through legal arguments before writing. Same price ($2/$8).

  • Zero backend changes — same API, just swapped the model string
  • Removed temperature parameter (o3 doesn't support custom values)
  • Legal citations jumped 7.8 → 9.0, field verification 8.6 → 9.4
  • e-OSCAR resistance initially dropped 8.8 → 7.6 (prompted v3 overhaul)
letter_generator.py autoresearch_prompts.py

Autoresearch Engine

Engine

Built Autoresearch Loop

Autonomous prompt optimization system — parses real credit reports, generates letters, scores them on 7 dimensions, and proposes improvements.

  • Parse: 68 accounts from 19 PDFs in corpus
  • Generate: Full dispute letters using each prompt pack
  • Score: 7 dimensions — e-OSCAR resistance, field verification, legal citations, deadline/consequences, DNR trap potential, personalization, professional tone
  • Improve: GPT analyzes weaknesses and proposes prompt rewrites
  • CLI: --pack, --sample, --iterations, --dry-run

  • DATABASE_URL env var with postgres:// → postgresql:// rewrite
  • SQLite-specific pragmas only run on SQLite connections
